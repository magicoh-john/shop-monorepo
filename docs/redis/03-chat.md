# 상담원 채팅과 Redis Pub/Sub

## 왜 필요한가

쇼핑몰에 상담원 채팅 기능을 추가한다고 가정합니다.  
채팅은 WebSocket으로 구현하는데, WebSocket 연결은 특정 서버에 종속됩니다.

서버가 한 대일 때는 문제가 없습니다.  
모든 사용자가 같은 서버에 연결되어 있으므로 서버 메모리에서 직접 메시지를 주고받을 수 있습니다.

```
[단일 서버]

사용자 A ──┐
           ├── 서버 1 (메모리에서 직접 중계) ──┬── 상담원
사용자 B ──┘                                   └── 사용자 A
```

**서버가 여러 대로 늘어나면 문제가 발생합니다.**

```
[다중 서버]

사용자 A → 서버 1 연결
상담원   → 서버 2 연결

서버 1과 서버 2는 서로의 연결을 모름
→ 사용자 A의 메시지가 상담원에게 전달되지 않음
```

---

## Redis Pub/Sub로 해결

Redis가 중간에서 메시지를 중계하면 서버가 몇 대든 관계없이 채팅이 가능합니다.

```
사용자 A → 서버 1 → Redis PUBLISH chat:room_1 "안녕하세요"
                           ↓
           서버 2 ← Redis SUBSCRIBE chat:room_1 수신
                           ↓
                        상담원에게 전달
```

모든 서버가 Redis의 채널을 구독하고 있으므로,  
어느 서버에 연결된 사용자라도 동일한 채팅방 메시지를 주고받을 수 있습니다.

---

## 단일 서버 vs 다중 서버

| 환경 | Redis 필요 여부 | 이유 |
|---|---|---|
| 서버 1대 | 불필요 | 메모리에서 직접 중계 가능 |
| 서버 여러 대 | **필수** | 서버 간 메시지 공유 수단이 없음 |

소규모 서비스를 단일 서버로 운영한다면 Redis 없이도 채팅을 구현할 수 있습니다.  
서비스가 성장해 서버를 추가(스케일 아웃)하는 시점에 Redis Pub/Sub가 필수가 됩니다.

---

## 구현 흐름

```
1. 사용자가 채팅방 입장
   → WebSocket 연결 (서버에 소켓 등록)

2. 사용자가 메시지 전송
   → 서버가 Redis PUBLISH chat:{roomId} 메시지

3. 같은 채널을 SUBSCRIBE 중인 모든 서버가 수신
   → 각 서버에 연결된 사용자에게 WebSocket으로 전달

4. 사용자가 채팅방 퇴장
   → WebSocket 연결 해제
```

---

## Redis가 명확히 필요한 이유

장바구니의 경우 "없으면 깜빡임이 생긴다"는 UX 문제였습니다.  
채팅의 경우 다중 서버 환경에서 Redis 없이는 **메시지 전달 자체가 불가능**합니다.

이것이 이 프로젝트에서 Redis를 도입하는 가장 명확한 이유입니다.
