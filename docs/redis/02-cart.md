# 장바구니와 Redis

## 현재 방식의 한계

현재 장바구니는 Zustand + localStorage로 관리합니다.  
localStorage는 브라우저에만 존재하기 때문에 서버는 그 데이터를 읽을 수 없습니다.

Next.js가 장바구니 페이지를 서버에서 미리 렌더링(SSR)할 때 장바구니가 빈 상태의 HTML을 만들고,  
브라우저에서 JavaScript가 실행된 후에야 Zustand가 localStorage를 읽어 화면을 다시 그립니다.  
이 때문에 처음에 빈 화면이었다가 상품이 뒤늦게 나타나는 **화면 깜빡임(Layout Shift)** 이 발생합니다.

### 직접 검증

브라우저 DevTools에서 JavaScript를 비활성화한 후 장바구니 페이지를 열면  
localStorage에 상품이 있어도 빈 화면이 표시됩니다.  
이것이 SSR 시점에 서버가 보내는 초기 HTML입니다.

---

## 회원 vs 비회원

| 사용자 | 장바구니 저장 위치 | SSR 가능 | 깜빡임 |
|---|---|---|---|
| 로그인 회원 | PostgreSQL (DB) | ✅ | 없음 |
| 비로그인 | localStorage | ❌ | 있음 |

로그인 회원은 서버가 DB에서 장바구니를 읽어 초기 HTML에 포함시키므로 깜빡임이 없습니다.  
**깜빡임 문제는 비로그인 사용자 전용 문제입니다.**

---

## 비로그인 장바구니를 Redis에 저장하는 이유

쿠키에 장바구니를 직접 저장하는 방법도 있지만, 쿠키의 용량 한계는 **4KB**입니다.  
상품 여러 개, 이미지 URL, 가격 등을 담으면 금방 초과합니다.

업계 표준 방식은 다음과 같습니다.

```
쿠키: session_id=abc123          ← 식별자만 저장 (작음)
Redis: cart:abc123 → [상품 목록]  ← 실제 데이터 저장 (서버)
```

쿠키에는 세션 ID만 저장하고, 실제 데이터는 Redis에 보관합니다.  
서버는 요청마다 쿠키의 세션 ID로 Redis에서 장바구니를 조회하므로  
SSR 시점에도 데이터를 읽어 초기 HTML에 포함시킬 수 있습니다.

---

## 로그인 시 장바구니 병합

비로그인 상태에서 담은 장바구니를 로그인 후에도 유지하려면 병합(merge) 로직이 필요합니다.

```
로그인 전:  cart:session_abc  → [티셔츠, 청바지]
로그인 후:  cart:user_123     → [운동화]

병합 결과:  cart:user_123     → [티셔츠, 청바지, 운동화]
            cart:session_abc  → 삭제
```

현재 Zustand 방식에는 이 병합 로직이 없습니다.  
Redis로 전환하면 로그인 시점에 두 장바구니를 자연스럽게 합칠 수 있습니다.

---

## 소결

비로그인 장바구니 하나만을 위해 Redis를 도입하는 것은 인프라 비용 대비 효과가 크지 않습니다.  
단, 이후 채팅 기능 등 Redis가 명확히 필요한 기능이 추가된다면,  
장바구니도 함께 Redis로 관리하는 것이 자연스럽습니다.
