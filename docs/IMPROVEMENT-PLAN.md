# 🚀 프로젝트 고도화 및 개선 방안 보고서

**문서 목적**: 현재 pnpm 모노레포 구조에서 발생하는 한계를 극복하고, 향후 프로젝트 확장(모바일, AI 모듈 추가 등)에 대비하여 시스템을 더욱 견고하고 효율적으로 만들기 위한 단계별 개선 로드맵입니다.

---

## 1. ⚡ 빌드 파이프라인 최적화 (Turborepo 도입)

**현재의 문제점**: 
프로젝트가 커지고 모듈이 많아지면, 특정 패키지(예: `apps/web`)의 코드만 수정했음에도 전체 모노레포의 린트(Lint), 테스트, 빌드가 다시 실행되어 CI/CD 배포 시간이 기하급수적으로 늘어날 위험이 있습니다.

**개선 방안: Turborepo 점진적 도입**
- Vercel에서 제공하는 모노레포 빌드 시스템인 **Turborepo**를 도입합니다.
- **스마트 캐싱(Smart Caching)**: 변경되지 않은 패키지의 빌드/테스트 결과는 이전 캐시에서 1초 만에 가져와 작업 시간을 획기적으로 단축합니다.
- **작업 오케스트레이션(Task Orchestration)**: 의존성 그래프를 분석하여 연관이 없는 패키지들의 빌드 작업을 병렬(Parallel)로 동시 실행하여 CPU 자원을 100% 활용합니다.
- **적용 시점**: `apps/mobile`이나 `apps/ai` 등 두 번째 애플리케이션 모듈이 추가되는 시점에 맞춰 루트에 `turbo.json`을 구성하여 도입하는 것을 권장합니다.

---

## 2. ⚙️ 환경 설정(Config)의 중앙화 및 모듈화

**현재의 문제점**: 
`tsconfig.json`, `eslint.config.js`, `.prettierrc` 등의 환경 설정 파일이 각 `apps/*`와 `packages/*` 하위에 개별적으로 존재할 경우, 팀원 간 코딩 컨벤션 파편화가 발생하고 일괄 수정 시 막대한 유지보수 비용이 듭니다.

**개선 방안: `packages/config` 패키지 분리**
- **설정 전용 패키지 신설**: `packages/eslint-config`, `packages/typescript-config` 등 설정만을 위한 공통 패키지를 만듭니다.
- **Extends 패턴 활용**: 각 앱과 패키지에서는 로컬 설정 파일에 복잡한 룰을 적지 않고, 아래와 같이 중앙 설정 패키지를 `extends`하여 사용합니다.
  ```json
  // apps/web/tsconfig.json 예시
  {
    "extends": "@my-monorepo/typescript-config/nextjs.json",
    "compilerOptions": {
      "baseUrl": "."
    }
  }
  ```
- **기대 효과**: 단일 진실 공급원(SSOT)을 설정 파일에도 적용하여 전체 워크스페이스의 코드 품질과 컨벤션을 100% 일관되게 통제할 수 있습니다.

## 3. 🛡️ 추가적인 아키텍처 고도화 과제

앞서 언급된 단점 극복 외에도, 실무 수준의 안정성을 확보하기 위해 아래의 개선 과제들을 추가로 제안합니다.

### ① Git Hooks 및 자동화 도입 (Husky + lint-staged)
- **내용**: 개발자가 코드를 커밋(Commit)하거나 푸시(Push)하기 직전에 강제로 ESLint, Prettier, TypeScript 타입 검사를 실행하도록 구성합니다.
- **기대 효과**: 에러가 있거나 컨벤션에 맞지 않는 코드가 중앙 저장소(GitHub)로 올라가는 것을 원천 차단하여 코드 리뷰 시간을 단축합니다.

### ② 공통 UI 컴포넌트의 완벽한 분리 (`packages/ui`)
- **내용**: 현재 `apps/web/components/ui`에 위치한 Shadcn UI 및 Tailwind 기반 공통 컴포넌트들을 `packages/ui`로 완전히 이관합니다.
- **기대 효과**: 향후 `apps/mobile`(React Native 등)이나 별도의 어드민 웹(`apps/admin`)을 만들 때, 버튼이나 입력 폼 같은 디자인 시스템을 그대로 가져다 쓸 수 있어 UI 일관성이 보장되고 개발 속도가 빨라집니다.

### ③ 자동화된 CI/CD 파이프라인 구축 (GitHub Actions)
- **내용**: PR이 올라오면 GitHub Actions가 자동으로 1. 패키지 설치 (`pnpm install`), 2. 타입 검사, 3. 단위 테스트, 4. 모의 빌드를 수행하도록 파이프라인을 작성합니다.
- **기대 효과**: 코드 결함이 있는 PR이 메인 브랜치에 Merge 되는 것을 방지하는 강력한 자동화 방어막을 구축합니다.
